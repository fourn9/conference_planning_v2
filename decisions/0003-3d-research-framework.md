# ADR-0003: 3D リサーチフレームワーク（規模 × 目的 × 職種）

- **Status**: Accepted
- **Date**: 2026-05
- **Tags**: research, methodology, persona, core-theme
- **Extends**: [ADR-0002](0002-research-source-evaluation.md)
- **Extended by**: [ADR-0004](0004-matrix-b-research-hardgate.md), [ADR-0005](0005-session-theme-structuring.md)

## Context（背景・問題）

ADR-0002（ソース評価 + 規模 × フェーズクロスカット）の上に構築しても、フレームワークは
依然として 3 つの構造的問題を見せた：

1. **同職種でも、目的によって関心が異なる**：「新規立ち上げ」フェーズの BE と
   「モダナイゼーション」フェーズの BE は別物。混ぜると平均化された誰にも刺さらないペルソナ。

2. **規模とアーキテクチャトピックの循環推論**：技術ブログのアーキテクチャトピックが
   企業を分類するのに使われていた — しかし技術ブログのコンテンツは企業が実際に何に
   投資しているかに依存する。ループ。

3. **課題／背景／理由／知りたいこと の混在**：単一フィールド「このセルの課題」が
   観察（何）、文脈（どこ・いつ）、解釈（なぜ）、探索アイデア（何を探究）を混ぜていた。
   検証可能性なし。

4. **「知りたいこと」解釈のドリフト**：エンジニアへのヒアリング質問として使う派と、
   主催者向けのセッションテーマ素材として使う派が混在。下流の異なる用途。

## Decision（決定）

**3 軸を 2 つの 2D マトリクスに分解** を採用：

### 軸

- **規模**（5 セグメント）：非 IT 大手 / IT 大手 / 成長期 / スタートアップ / PMF 前
- **目的**（6 レベル、P1-P6）：新規立ち上げ / スケール / マルチプロダクト統合 /
  モダナイゼーション / AI 組み込み / AI ネイティブ再構築
- **職種**（高優先 7）：BE/FS / TL / EM / SRE / インフラ / セキュリティ / データ

### 2 つの 2D マトリクス

- **Matrix A**：規模 × 目的（事業戦略レイヤー、IR ／戦略／経営層から得る）
- **Matrix B**：職種 × 目的（技術実装レイヤー、技術ブログ／登壇／エンジニアリング
  ポッドキャストから得る）

両方が **目的軸** を共通の背骨として共有。

### Matrix B のセル構造（必須）

各セルに 4 つの番号付きセクション：

1. **課題（What）**：観察されたギャップ
2. **背景（Where / When / How）**：事実としての文脈
3. **理由（Why）**：仮説された原因
4. **知りたいこと（= セッションテーマ素材）**：登壇者向けの抽象トピック

セクション間で同じ番号付け（課題 1 ↔ 背景 1 ↔ 理由 1 ↔ 知りたい 1）。

### 「知りたいこと」= セッションテーマ素材

ヒアリング質問ではない。登壇者リサーチと CFP 応募者に適した抽象トピック。

## Consequences（結果・影響）

### Positive
- ペルソナの深さ：同職種・異目的の関心を区別可能に
- ソースレイヤー分離で循環推論を防ぐ
- 事実／仮説分離で検証可能性
- セッションテーマ素材が特定のセルにトレース可能
- カンファレンス領域を超えて再利用可能

### Negative / Trade-offs
- 42 Matrix B セル × 4 セクション = 著作コストが相当
- 背景／理由フィールドは事実収集と仮説形成が必要
- Matrix A の IR リサーチは技術ブログスキャンより遅い

### Risks
- P5 vs P6 境界は最も間違えやすい（AI 組み込み vs AI ネイティブ）
- セルごとの「該当する規模」アノテーションが Matrix A とクロスチェックなしには
  主観的になり得る
- スタートアップ時間枠延長が他所に適用されるリスク

## Alternatives Considered（検討した代替案）

### 代替案 A：3D マトリクス（5×6×7=210 セル）

却下 — 認知負荷が高すぎる。2 つの 2D マトリクスがメンタルトラクタビリティを保つ。

### 代替案 B：規模 × 職種のみ（ADR-0002 の延長）

却下 — この ADR を動機付けた「同職種・異目的」の洞察を逃す。

### 代替案 C：規模 × 目的のみ（職種は後にペルソナオーバーレイ）

却下 — ペルソナ作成時に職種を貼り付けると、職種固有の懸念が粒度を失う。

### 代替案 D：ドライバー軸を主軸に（目的の代わりに）

[事業成長] / [規制] / [技術負債] / [AI 統合] を主軸に。却下 — ドライバーと目的は
重複し、ドライバーは理由フィールドでより良く捉えられる。
