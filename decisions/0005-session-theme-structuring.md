# ADR-0005: セッションテーマ構造化方法論（コア → 大／中／小テーマ）

- **Status**: Accepted
- **Date**: 2026-05
- **Tags**: methodology, session-theme
- **Extends**: [ADR-0003](0003-3d-research-framework.md), [ADR-0004](0004-matrix-b-research-hardgate.md)

## Context（背景・問題）

Phase 4（ヒートマップ + audience-shape クラスタリングでのコアテーマ抽出）後、
暗黙のギャップが残った：コアテーマはどうやって CFP 応募者と登壇者が自分を位置付けられる
構造になるのか？

実際のカンファレンス企画ワークフローでは、このギャップは繰り返しの反復で埋められていた：

- 複数の軸選択が試行された（時間 / レイヤー / 判断 / 対立 / フェーズ / 主体 / 階層）
- 批判的見直しループがバイアス（AI 過剰表現、均等粒度、二項対立トラップ）を浮上させた
- 大テーマ間の粒度非対称が繰り返し議論された

これを明示しないと、フレームワークを再利用するすべてのカンファレンスがこれらの
パターンを苦労して再発見することになる。

## Decision（決定）

Phase 4（コアテーマ抽出）と Phase 5（登壇者リサーチ）の間に **Phase 4.6** を採用、
8 サブステップで：

### Phase 4.6 サブステップ

1. **入力レビュー + 焦点設定**（キャッチコピー候補を含む）
2. **大テーマ仮設定（3〜5）** — 通常 Why/What、How、Who、その他
3. **粒度の非対称性を許容** — 戦略 ≈ 3 中、実装 ≈ 5-6 中、組織 ≈ 4 中
4. **中テーマ軸選び** — 複数試行（時間 / レイヤー / 判断 / 対立 / フェーズ / 主体 / 階層）
5. **小テーマ = 議論ポイント**、ラベルではない
6. **批判的見直しループ ×2 最低** — 粒度／ MECE ／固有名詞／時代性／ターゲット整合／
   二項対立／議論-vs-ラベル／ AI バイアス
7. **抜け論点チェック** — 過去データ、または隣接カンファレンス、または Matrix B 集計
8. **横軸 + キャッチコピー最終化**

### 鍵となる opinionated な選択

- **粒度非対称は意図的**：異なる大テーマは異なる自然な中数を持つ。均等を強制すると
  不自然なバケットを生む。
- **小テーマは議論ポイント**：「サービス境界の引き直しとモジュラモノリスへの揺り戻し」 —
  「マイクロサービス」ではなく。
- **批判的見直し ×2**：1 回目は表面の問題を捉え、2 回目は 1 回目を生き残ったバイアスを捉える。
- **過去データなしも OK**：Matrix B 集計（スキル：`analysis/matrix-b-genre-aggregation/`）と
  隣接カンファレンス分析（`analysis/adjacent-conference-analysis/`）が代替。

## Consequences（結果・影響）

### Positive
- 暗黙の「テーマを構造化」ステップが明示的かつ再利用可能になる
- 批判的見直しループが 2 パスレビューを強制（2 回目のバイアスを捉える）
- 粒度非対称が事前に受け入れられ、「これらのカウントが揃わない」議論の繰り返しを除去
- 「過去データなし」パスが文書化される（初回イベントを完全サポート）

### Negative / Trade-offs
- Phase 4.6 がタイムラインに 2〜3 日追加
- 軸選択に単一の正解はない（複数試行）
- 「非対称粒度」はチームの明示的な合意が必要

### Risks
- 批判的見直しがプロセス化することがある（チェックリストを機械的に適用するだけで
  真に疑問視しない）；エージェントはチェックリストを機械的に適用するのではなく
  「本当にそうか？」を問うべき
- 「議論ポイント」はチームの書く規律が必要（ラベルへのドリフトがデフォルト）

## Alternatives Considered（検討した代替案）

### 代替案 A：コアテーマが直接登壇者リサーチへ

却下 — 構造なし、CFP と登壇者打診が曖昧。

### 代替案 B：大 → 中 → 小を均等粒度で

却下 — 異なる性質の大テーマ（戦略 vs 実装 vs 組織）に不自然なバケットを強制。

### 代替案 C：単一軸構造（例：すべて目的で組織化）

却下 — 目的だけでは組織テーマ、設計思想 等を捉えられない。

### 代替案 D：登壇者リサーチ後のテーマ構造化

却下 — 順序を逆にする；登壇者がテーマに合致すべき、逆ではない。
