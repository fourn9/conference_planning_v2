# 失敗パターン集

このフレームワークを使うとき避けるべき 20 のパターン。スキルや ADR からクロスリファレンスできるよう番号付き。

---

## リサーチフェーズの失敗

### #1 — 技術記事レンズで Matrix A を作る

**症状**：「技術記事で議論されている内容」から Matrix A を作る

**結果**：事業戦略と技術トレンドを混同し、規模ごとの主戦場を見誤る

**対策**：Matrix A は必ず IR・中期経営計画・経営層インタビューから始める。
技術記事は補強証拠であって、主軸ではない。

### #2 — 課題／背景／理由／知りたいことを Matrix B 内で混在させる

**症状**：1 つのセルが、観察された事象、文脈、仮説、探索アイデアを混ぜている

**結果**：事実と仮説の区別がない、検証可能性が落ちる

**対策**：4 セクション（課題／背景／理由／知りたいこと）に番号付きで対応させる。
Phase 3 スキル参照。

### #3 — 「知りたいこと」をヒアリング質問形式で書く

**症状**：「LINE ヤフーで何をしましたか？」のような質問形式

**結果**：セッションテーマに直接変換できない

**対策**：「知りたいこと」は **ターゲットがイベントで聞きたい話＝セッションテーマ素材**
として書く。質問はヒアリング設計時に別途作る。

### #4 — 特定企業の事例名で「知りたいこと」を書く

**症状**：「LINE ヤフー型 -1→1 新規開発の実装ディテール」

**結果**：トピックの抽象度が低く、他企業が話せない

**対策**：「-1→1 型新規開発のアーキパターン」のように抽象トピックで書く。
具体企業は登壇者リサーチ時に紐付け。

### #5 — 単一規模のセッションをコアに据える

**症状**：1 規模だけに刺さるトピック（例：非 IT 大手限定の SAP 2027）をコアテーマにする

**結果**：聴講者層の大半に刺さらない

**対策**：必ず Matrix A で「複数規模が ◎」を確認してからコアに据える。

### #6 — 番号付き縦対応を崩す

**症状**：課題 3 項目、背景 4 項目、理由 2 項目、知りたいこと 5 項目のような不揃い

**結果**：どの行がどのトピックの話か追跡できない

**対策**：必ず同数の番号で揃える。空欄になる箇所は「該当少」と明示。

### #7 — 機械的クラスタリングで満足する

**症状**：Active P-set の重なりだけでクラスタを作り「綺麗に 3 つに割れた」で終わる

**結果**：実際の聴衆構造を捉え損ね、企画上の解像度が出ない

**対策**：Phase 4-b トピック起点ヒートマップ＋ 4-c audience-shape クラスタリングを必ず通す。
「3 つに割れた」と見えたら一度疑う。

### #8 — 役割の近さで職種ペアを束ねる

**症状**：「TL ↔ EM は文化変革で近い」「SRE ↔ インフラは技術領域が近い」と直感で束ねる

**結果**：実際の課題コードと一致せず、セッション設計時に話がかみ合わない

**対策**：**「同じ課題コードを別角度で見ているか」** で束ね直す。

### #9 — 内部検証なしに Matrix B を確定する

**症状**：外部公開情報だけで Matrix B を完成版として登壇者リサーチに進む

**結果**：「Web アプリ前提」「マイクロサービス推奨前提」等の暗黙バイアスが残る

**対策**：Phase 4.5 で必ず内部エンジニア 2〜4 名にヒアリングし、不足軸を補ってから
登壇者リサーチに進む。

### #10 — 仮説を抽象的すぎる表現で内部質問する

**症状**：「P1〜P6 のどれですか」「課題は何ですか」だけを聞く

**結果**：相手から「抽象的すぎてイメージしづらい」と返され、具体性のある回答が得られない

**対策**：質問時に **(a) P 軸の定義をコピペで添える (b) Matrix B の課題仮説を抜粋して提示
(c) 「実感と合うか・ズレるか」「むしろこっちの方が気になる論点は？」と並べて聞く**
の 3 点セットで投げる。

## Matrix B ハードゲートの失敗（ADR-0004）

### #11 — 手法名キーワード検索で Matrix B を埋める

**症状**：「ADR」「ASDD」「DDD」等の手法名キーワードで検索し、ヒットした記事を採用する

**結果**：概念解説・ハンドブック・参加レポ・学習記事ばかりが集まり、当事者の現場アーキ
判断は得られない

**対策**：G3（自社プロダクトでの判断）と G7（検索の深さ）を必ず通す。手法名ではなく
「人物・事象起点で本文を読み込んでから採否判定」する。

### #12 — 職種カテゴリの「縁」が壊れる

**症状**：VPoE／アーキテクト／ML エンジニア／スタートアップ CTO 等、Matrix B 対象 7 職種に
収まらない隣接職種の記事を、BE/FS や TL 行にそのまま入れてしまう

**結果**：同じ著者が「TL 行では OK、BE/FS 行では NG」になる構造的な行間齟齬が発生

**対策**：隣接職種ルールに従い、行ごとに G5 厳密適合をチェック。前提知識枠への振り分けも併用する。

### #13 — 会社単位でフェーズを判断する

**症状**：「Sansan はメガベンチャー」のように会社単位で P 帰属を判定し、
事業ユニット別の違いを無視する

**結果**：同じ会社内でも事業ユニットごとにフェーズが異なるのに、平均化された誤帰属が起きる

**対策**：ハードゲート G2「**所属プロダクト（事業ユニット粒度）** のフェーズが特定可能」を
必ず通す。会社単位ではなくプロダクト単位で再判定する。

### #14 — 業界トレンド・他社事例を自社事例として誤採用

**症状**：「業界の REST API トレンド」「Backend for Agent の概念整理」のような
業界解説・他社事例まとめを、当事者の自社判断と混同して採用する

**結果**：「現場で何を判断したか」の一次情報が不足し、コアテーマ抽出の根拠が弱くなる

**対策**：ハードゲート G3「**自社プロダクトでの** アーキ判断・実装」を必ず通す。
業界トレンド系の有用な記事は前提知識枠に分離する。

### #15 — 判定基準にフェーズ課題・組織体制・経営宣言を混ぜる

**症状**：判定基準を書くときに、フェーズ課題（「ADR 運用形骸化」「自律と統制」）／
組織体制（「専任 SRE を置く余裕がない」）／アーキ前提決め打ち（「数十単位のマイクロサービス」）／
経営宣言（「AI-First 宣言」）を混ぜて記述する

**結果**：claude.ai が「このプロダクトがそのフェーズに該当するか」を **外部から客観判断する材料**
として使えない

**対策**：判定基準は **3 要素（サービス属性／プレス・外部表記／アーキシグナル）** で構成。
組織体制／アーキ前提決め打ち／フェーズ課題／経営宣言は判定基準に入れない。

## セッションテーマ構造化の失敗（ADR-0005）

### #16 — 全テーマで抽象度を揃えようとする

**症状**：戦略・実装・組織のすべてで「中テーマ数」「小テーマ数」を機械的に揃える

**結果**：戦略系（抽象化可能）と実装系（専門領域分かれ）と組織系（階層構造）の
本質的な性質差が消え、無理に揃えた粒度が不自然になる

**対策**：各大テーマの本質に合わせて粒度を非対称に許容する
（戦略 ≈ 3／実装 ≈ 5-6／組織 ≈ 4）。

### #17 — 1 テーマ = 1 セッションと考える

**症状**：「中テーマ数 = セッション数」と捉え、テーマを増やすか減らすかをセッション数で判断する

**結果**：テーマ構造化が「セッション割り当て」に矮小化され、CFP の応募者振り分け軸・
登壇者リサーチのグルーピング軸として機能しなくなる

**対策**：テーマは **整理用** であり、1 テーマから複数セッションが派生してよい。
「Why/What・How・Who で切る」枠組みを優先する。

### #18 — AI 偏重で他論点が薄まる

**症状**：AI の重要性に引きずられ、すべての中テーマに AI を関連させようとする／
AI を独立大テーマにしすぎる

**結果**：「AI 以外の論点（レガシー刷新・組織論・規模拡大の継続痛み）」が薄くなり、
ターゲット（メガベン以上の TL/EM）が「うちは違う」と離れる

**対策**：「AI 偏重チェック」で見直す。AI を 1 つの論点として扱い、戦略系の中テーマで
「未来・現在・過去〜未来」のバランスを取る。

### #19 — 二項対立で切って取りこぼす

**症状**：「広げる vs 統合する」「自律 vs 統制」のような二項対立で中テーマを切る

**結果**：「あえて分けたままにする」「ROI で維持を判断する」のような第三の選択肢が抜ける

**対策**：複数軸候補を出し、三つ巴・スペクトラム型の切り方も検討する
（例：広げる／つなぐ／分ける／取り込む の 4 方向）。

### #20 — 「やっていること」の列挙で論点にならない

**症状**：小テーマが「マイクロサービス化」「データ基盤」のように、やっていることの名前だけになる

**結果**：「実行時に議論されるポイント」が見えず、CFP 応募者・登壇者が何を話せばよいか不明

**対策**：OK/NG 例に従い、「サービス境界の引き直しとモジュラーモノリスへの揺り戻し」
「既存 DB からの段階的脱却」のような **議論ポイント** で書く。
