# スキル：失敗からのプロンプト改善

## 目的

`research/review-output` で見つけた問題を、**次に投げるプロンプトの具体的な修正**に変換する。
「成果を見て直す」をその場の感覚でなく、`failure-patterns.md`（#1–#20）への対応として行い、再現性を持たせる。
`research/design-prompt` → 実行 → `research/review-output` → 本スキル、で1周するループの締め。

## 手順

### 1. 問題を失敗パターンにマップ
レビューの指摘を `failure-patterns.md` の番号に紐づける。例：
- 概念解説ばかり採用された → #11（手法名キーワード検索）
- 隣接職種が混ざった → #12（カテゴリの縁が壊れる）
- 会社単位でフェーズ判定 → #13
- 業界トレンドを自社事例化 → #14
- 数合わせの水増し → （カバレッジ/確定数の分離不足）

### 2. パターン → プロンプトの具体修正
各失敗に対し、`research/design-prompt` の原則のどれを強化するか決めて**プロンプトを書き換える**：
- NG パターンを「検索戦略」に明示追記する
- 該当ゲート（G1-G7）の判定を厳格化・文言追加
- 「前回失敗の文脈」セクションに**今回の具体例**を入れて再発を抑える
- 件数要件・保留リスト上限を調整（カバレッジは緩めない）

### 3. 再リサーチ
改訂プロンプトで再実行 → 再び `research/review-output`。

### 4. 反復の停止条件
- 必達カバレッジを満たし、かつ**連続して新規問題が出なくなった**ら停止
- 単純な「確定数 N 件」では止めない（尾を取りこぼす）

## 学習への接続（meta）

- 同じ修正を**2回**したら → `/draft-adr`（規律として固定する価値）
- 反復手順がスキル化できそうなら → `/draft-skill`
- 単発の非自明な気付きは → `/record-learning`
- 既存スキル/テンプレの不備に気づいたら → `/patch-skill`（`pinned: true` は直接編集せず記録して提案）

## アンチパターン

- レビューの指摘を**プロンプトに反映せず**口頭メモで済ませる（次セッションで再発する）
- カバレッジを犠牲にして「問題を減らす」（探索が浅くなるだけ）
