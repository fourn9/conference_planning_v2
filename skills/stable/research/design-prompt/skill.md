# スキル：ディープリサーチ用プロンプト設計

## 目的

claude.ai 等の LLM リサーチに投げる**プロンプトそのものを設計する**スキル。
「広く探す」と「捏造しない」を同時に成立させる。Matrix A/B の参考URL収集（Phase 2・3）や
登壇者・事例・競合リサーチなど、外部 LLM に深掘りを任せる場面で使う。

`research/review-output`（成果レビュー）→ `research/iterate-from-failures`（改善）と1本のループを成す。
具体テンプレは `methodology/templates/claude-ai-prompt-matrix-b.md` を出発点にできる。

## 設計原則（これを満たすプロンプトを書く）

### 1. ノルマの掛け先を分ける（最重要）
「確定 N 件必達」を素で課すと、実在候補が尽きたとき**数合わせのための推測採用・弱い根拠の水増し・人物/記事の創作**に流れる。掛け先を3つに分離する：
- **探索カバレッジ＝必達**：規定の経路・対象範囲・全セルを走査し尽くす（ここは妥協しない）
- **確定数＝目標（捏造禁止）**：目標値は置くが、足りなければ探索を広げ、なお不足なら**実数を正直に報告**する
- **保留リスト＝breadth の受け皿**：広く拾ったが裏取り未了の候補を捨てず上限つきで記録

### 2. 実在性 > 数（捏造の絶対禁止）
実在しない人物・記事・URL の創作は重大違反、と明文化する。「該当なし」を許容する。

### 3. 本文確認を必須にする
URL を開いて**本文を読んでから採否**。スニペット判断を禁止。アクセス確認日を記録させる。

### 4. closed list を作らない
**固有名（企業・人・自社の過去イベント・ベンダー資料）を「例示」や「探索ソース」としてプロンプトに埋め込まない。** それ自体が事実上の閉じたリストになり探索が偏る（前回登壇者が再浮上して新規発掘が死ぬ等）。判定は**ゲート＋客観基準**でやらせ、固有名は置かない。自社の過去イベントを探索ソースにするのは循環参照なので外す。

### 5. 複数経路で探索させる
単一キーワード一致に頼らない。最低2経路（例：対象起点の網羅クロール＋事象/論点起点の検索）を指示する。

### 6. 出力を分割させる
採否チェック表（落選含む）／採用一覧（転記可能形式）／補足枠（基準を一部満たさないが文脈に有用）／**「該当なしセル」一覧**（＝仮説見直しフラグ）に分ける。

### 7. プレースホルダ化して再利用可能にする
対象・件数要件・判定基準を `[...]` で外出しし、別案件へ転用できる形にする。

## 手順

1. リサーチの**目的と対象**を1文で固定（何を集めるか／誰の何を裏付けるか）
2. **採否ハードゲート**を定義（`research/hardgate-evaluation` の G1-G7 を土台に、案件に合わせて調整）
3. 上記原則1-7をプロンプトに織り込む（特に原則1・4は明示セクションにする）
4. **出力フォーマット**を指定（§6 の分割）
5. 既知の `failure-patterns.md` から該当しそうな失敗の**NG 節**をプロンプトに先回りで入れる
6. テンプレ（`methodology/templates/claude-ai-prompt-matrix-b.md`）があれば差し替えて使う

## 検証（投げる前の Exit Criteria）

- [ ] カバレッジ必達／確定数目標／保留リストの3分離が明記されているか
- [ ] 「実在しないものの創作は違反」「該当なし許容」を書いたか
- [ ] 固有名を例示・探索ソースに埋め込んでいないか（closed list 禁止）
- [ ] 本文確認必須・アクセス日記録を指示したか
- [ ] 出力に「該当なしセル一覧」を含めたか
- [ ] プレースホルダ化して再利用できる形か
