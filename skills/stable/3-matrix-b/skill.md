# スキル：Phase 3 — Matrix B 作成（職種 × 目的）

## 目的

*職種 × 目的* の 2D マトリクスを構築する — 42 セル（7 職種 × 6 目的）、
各セルに 4 セクション構造（課題 / 背景 / 理由 / 知りたいこと）を当事者エンジニアの
アウトプットから引き出す。

## このスキルを呼び出すとき

- Phase 1 と Phase 2 が `completed`
- `conference-planning.yml` に `phase_3_matrix_b: pending`

## 手順（エージェント向け）

### Step 3.1：職種別の claude.ai リサーチプロンプト

職種ごとに 1 つのプロンプトを生成（7 職種で計 7 プロンプト）。エージェントは以下のテンプレを使う：

```
# [カンファレンス名] — Matrix B リサーチ：職種 [職種名]

## 目的
[職種名] について、P1-P6 目的にわたって最もよく言及される現在の課題、文脈、
「知りたいこと」を、当事者エンジニアのアウトプットから特定する。

## ソース優先順位（高い順）
1. 当事者の本人発信（個人ブログ、登壇者本人としてのスライド、ゲスト出演ポッドキャスト）
2. 企業技術ブログで著者明示あり
3. 書籍 / 登壇
4. 業界レポート（最終手段、その旨明示）

## ソース採用のハードゲート（G1-G7）
G1：著者の職種が明示的に識別可能（[職種名] に一致）
G2：アウトプットが 2024-01 以降
G3：内容が当事者経験を反映（概念記事ではない）
G4：URL が検証可能で引用が再現できる
G5：[職種名] に厳密適合（隣接職種はこのセルでは不可）
G6：タイトルを超えたコンテンツ（実質、ブランディングではない）
G7：目的カテゴリと文脈的一致（キーワードではない）

## セルごと（P1〜P6）の出力構造
課題（What — 観察された事象・ギャップ）：
1. ...
2. ...
3. ...

背景（Where / When / How — 事実としての文脈）：
1. （課題 1 の背景）
2. ...
3. ...

理由（Why — 仮説）：
1. ...
2. ...
3. ...

知りたいこと（= セッションテーマ素材）：
1. ...
2. ...
3. ...

該当する規模：Matrix A から判定

参考 URL（G1-G7 通過のみ）：
- [タイトル] | YYYY-MM-DD | URL | 著者 / 引用

## 重要なルール
- 「該当者なし」は許容。弱いソースで水増ししない。
- 1 記事 = 複数「知りたいこと」（M:N）可。
- タイトルではなく記事の引用を明示。
- 言及される企業は例示であり、closed list ではない。
```

職種ごとに 1 プロンプト生成。実行は Claude Code 内のディープリサーチワークフローが第一選択（ADR-0007）。外部 LLM（claude.ai Deep Research）への貼り付けはフォールバック。

### Step 3.2：職種別アウトプットを受領・統合

各職種（7 職種）について：

1. 生アウトプットをユーザーの作業ディレクトリに保存
2. ハードゲートフィルタを適用（`skills/stable/research/hardgate-evaluation/`）
3. `templates/matrix-b.md` セル構造（セルあたり 4 セクション）にフォーマット
4. Matrix A に対して該当規模をクロスチェック

生リサーチからの採用率は通常 15-25%。これは正常。

### Step 3.3：ジャンル分布チェック（早期警告）

全 42 セルにわたる「知りたいこと」を集計。ジャンル別に集計（AI / モダナイゼーション /
マルチプロダクト / データ / 可観測性 / セキュリティ / 組織）。

AI が全エントリの 60% を超えたら：ユーザーに警告。Phase 4 で意識的に多様化が必要。
