# スキル：Phase 4.5 — 内部検証

## 目的

Matrix B とコアテーマ仮説を、ターゲット聴衆の実際のエンジニアに対して検証する。
外部由来だが内部経験と合わない仮説を捉える。

## このスキルを呼び出すとき

- Phase 3 と Phase 4 が `completed`
- `conference-planning.yml` に `phase_4_5_internal_validation: pending`
- ユーザーがターゲット聴衆に合うエンジニア 2〜4 名にアクセスできる

## 手順（エージェント向け）

### Step 4.5.1：選定基準

ユーザーが 2〜4 名の内部エンジニア（または内部不可なら外部のターゲット適合
エンジニア）を選ぶのを支援：

- ターゲット聴衆の規模セグメントと職種カテゴリに合致
- 多様な職種（BE/FS、SRE、データ 等）で広く
- 構造化された DM に応じる意思がある

### Step 4.5.2：初回 DM テンプレート

エージェントがユーザー送信用の DM を生成。テンプレ：

```
[エンジニア名] 様

[カンファレンス名]（[ターゲット日]）の企画を進めています。目的は
[一文のカンファレンスゴール]。

企画フェーズにいて、[職種] のエンジニアが実際に何を気にしているかについての
仮説を検証したいです。あなたの意見を聞かせてください。

カンファレンスについて：
[2〜3 行]

検証したい仮説（公開ソースから構築、外れている可能性あり）：
[Matrix B から該当職種 × 想定目的の「知りたいこと」を 3〜5 抜粋]

3 つの簡単な質問：

1. 下記の P1〜P6 のうち、現在のプロダクトフェーズに最も近いものは？
   （複数可、「該当なし」も OK）
[axes.md から P1-P6 を挿入]

2. 実際にぶつかっている課題と、その背景文脈は何ですか？

3. 上記の仮説のうち、実感と合うもの／ズレるものはありますか？
   または、リストにない「むしろこちらの方が気になる」課題はありますか？

長文は不要、1 段落でも非常に助かります。

ありがとうございます！
```

### Step 4.5.3：回答を受領

各回答について：

1. 自己定位の目的を抽出（P1-P6）
2. 述べられた課題 + 背景
3. 送った仮説それぞれを確認 / 反証
4. 仮説に含まれていなかった「実際に気にしていること」の新規追加

### Step 4.5.4：パターンマッピング

各回答について問う：エンジニアのメンタルモデルが既知パターン（Data Mesh、
モジュラモノリス、Conway の法則 等）に合致するか？

エンジニアはしばしばパターン名を口にしない。**構造で照合**、名前ではない。
構造的一致が見えたら、第 2 ラウンド DM で確認。

### Step 4.5.5：第 2 ラウンド DM テンプレート（パターン確認）

```
[エンジニア名] 様

詳細な考えをありがとうございました！

説明を読んで、構造的に [パターン名] と重なるところに気付きました：
- [X] に関するあなたの指摘は [パターンの原則 1] にマップ
- [Y] に関する懸念は [パターンの原則 2] にマップ

確認したいのですが：[パターン名] はチームの議論で候補方向として視野にありますか？
または、別の方向（中央集権を維持、別のアーキテクチャ等）を考えていますか？

これで [パターン] を独立トラックとして立てるか、[隣接トピック] 内に含めるかが
クリアになります。

ありがとうございます！
```

### Step 4.5.6：Matrix B v2 に統合

各回答者について：

1. セルごとに仮説 4 セクションを確認または調整
2. 提起された不足角度を追加
3. 「内部エンジニア X（合意済み）で検証」とメモ

`templates/matrix-b.md` を v2 に更新。

### Step 4.5.7：状態を更新

```yaml
phases:
  phase_4_5_internal_validation:
    status: completed
    notes: "[X] 名のエンジニアで検証。[Y] の不足角度を追加。[Z] 仮説を修正。"
```

`LEARNINGS.md` にログ。

## 使うテンプレート

- 上記の DM テンプレート
- `templates/matrix-b.md`（v2 更新用）

## 検証

- [ ] 2〜4 名の回答者（1 名だけではない）
- [ ] 職種多様性（BE/FS、SRE、データ 等）
- [ ] 各回答者に新規角度追加の機会
- [ ] パターンマッピングを構造で実施、名前ではない
- [ ] 該当する場合、第 2 ラウンド確認 DM を送信

## 注意すべき失敗パターン

- 1 名のエンジニアだけで検証（単一人物バイアス）
- 抽象的すぎる質問（#10） — エンジニアが「P1-P6 だけ」に応えるのに苦労する。
  上のテンプレは仮説を含めて文脈提供
- パターンが浮上したのに第 2 ラウンド DM をスキップ（精度を失う）
