## 採否ハードゲート 7 条件（全条件を満たすもののみ採用）

正式定義は `skills/stable/research/hardgate-evaluation/`（ADR-0004）。番号・意味はそれに準拠する。
本文確認時に1件ずつ ✓/✗ をチェックし、出力に明示すること。

- [ ] **G1. 識別可能性**：執筆者の氏名・職種・現所属が特定可能（明示／プロフィール／登壇資料から。匿名・帰属不明は不可）
- [ ] **G2. 直近性**：公開日が 2024-01 以降（PMF 前／非常に早期の企業のみ 2021 年以降も許容。イベント日に応じて調整）
- [ ] **G3. 当事者経験**：内容が**「自社プロダクトでの」**アーキ判断・実装を含む（業界トレンド解説・他社事例まとめ・概念紹介は不可）
- [ ] **G4. 検証可能性**：URL が実在しアクセス可能、著者が実在、引用が本文から再現できる
- [ ] **G5. 厳密カテゴリ適合**：職種カテゴリへの厳密適合（隣接職種は当該行に入れない）
- [ ] **G6. タイトルを超えた実質**：本人の具体的な主張・判断が本文にある（肩書・登壇者バイオ・プレスリリースだけは不可）
- [ ] **G7. 文脈一致**：本文全体の文脈がテーマ・目的に一致（キーワード単純一致での表層採用・タイトル／サービス名の誤読は不可）

## プロセス規律（厳守）

1. **本文を必ず読み込む**：URLを開き、本文を読んでから採否判定。スニペットだけで判断しない
2. **3点メモを取る**（各候補ごと）：
   - (i) 著者は誰か（職種・所属プロダクト・関与年数）
   - (ii) 内容種別（自社判断 / 業界トレンド / 概念解説）
   - (iii) どの「知りたいこと」の何番を、本文のどの段落が裏付けるか（**引用付き**）
3. **Matrix A は判定基準の材料として使う**：記載企業は**代表例にすぎない**。Matrix A 未収録の企業を最低 [L社] 含めること
4. **1記事 = 複数の知りたいこと番号を裏付けてよい**（M:N 関係で記録）
5. **「該当記事なし」を許容**：無理にセルを埋めず、ゼロのセルは Matrix A 仮説の見直しフラグとして明示
6. **実在確認の厳格化**：URL を1件ずつ開き、本文確認＋アクセス確認日を記録
7. **フェーズ判定はプロダクト（事業ユニット）単位**：会社単位で判定しない。所属プロダクトのフェーズ（P1-P6）が特定できない候補は採用しない

## 検索戦略

### 推奨アプローチ
- **対象プロダクト起点の探索**：
  - 自社技術ブログ（直近12ヶ月）の網羅クロール
  - エンジニア個人の発信（note・Zenn・X 等）
  - カンファレンス登壇資料（speakerdeck・各種カンファ後の振り返り記事）
  - エンジニア採用ページ（職種・組織図のヒント）
- **事象起点の検索**（例：「マイクロサービス分割」「共有DB脱却」「Backend for Agent 実装」「モノリス回帰」「-1→1 開発」）

### NG（前回失敗パターン）
- **手法名キーワード単独検索**（ADR／ASDD／DDD 等）→ 概念解説・ハンドブックに偏る
- **「企業規模 × AI」のような表層キーワード一致採用** → 内容と乖離した記事を拾う（例：プラン記事・サービス名誤読）
- **ハンドブック・解説サイト・Qiita のまとめ記事** → 当事者の現場判断ではない

## 規模×フェーズ判定基準（Matrix A から抽出）

[Claude Code が Matrix A 本文から事前抽出して挿入]

**判定基準の構成原則**（ADR-0004）：外部から客観的に確認できるプロダクト属性で構成する。
**組織体制**（チーム規模・専任有無）／**アーキ前提決め打ち**（「数十単位のマイクロサービス」
「共有DB依存」「単一プロダクト前提」等）／**フェーズ課題・論点**（Matrix B のセル内容）／
**経営宣言**（AI-First 宣言等）は判定基準に入れない。

各フェーズは以下3要素で記述：
- **サービス属性**：リリース経過時間・ユーザー規模・バージョン状態・機能展開
- **プレス/外部表記**：プレスリリース・採用ページ・登壇資料で確認できるシグナルワード
- **アーキシグナル**：記事から推定できる構成（決め打ちはしない）

**1プロダクト=複数フェーズ該当を許容**（例：新規 AIネイティブ立ち上げは P1 と P6(B) 両方該当）。

例（P2 既存サービス拡張・スケール）：
- サービス属性：リリース後1年以上経過、GA 状態、ユーザー数 数千〜数百万規模、機能を継続追加中
- プレス/外部表記：「機能追加」「○○万ユーザー達成」「海外展開」「新プラン提供」
- アーキシグナル：稼働中の既存アーキで機能追加・性能改善（モノリス／モジュラモノリス／MS いずれもありうる）

## 隣接職種ルール

Matrix B 対象7職種（BE/FS, TL, EM, SRE, インフラ, セキュリティ, データ）に対する隣接職種の取り扱い：

| 隣接職種 | 配置基準 |
|---|---|
| VPoE | EM 行（または独自セクション） |
| アーキテクト | TL 行（職務範囲が重なる場合のみ） |
| データエンジニア／データサイエンティスト／データプラットフォームエンジニア | **データ職種行 OK**（Matrix B のデータ行に統合済み） |
| MLエンジニア | 前提知識枠（P 別、特に P5/P6） |
| スタートアップCTO（数名〜30名規模、TL兼任明示） | TL 行 OK |
| メガベンチャー以上の CTO | **Matrix B 対象外**（Matrix A・コアテーマ CT-2/3/4・CTO ペルソナで扱う） |

**重要**：同じ著者が「TL 行ではOK、BE/FS 行ではNG」になりうる。**行ごとに G5 厳密適合**をチェック。

## 厳守ルール

- 出典URLは**実在確認**したものだけ（必ずアクセスして本文を確認する）
- 著者が特定できない記事は**推測で職種を補わず、不採用**にする
- 手法名解説記事（ADR・ASDD・DDD 等）は**採用URL欄に入れない**（補足解説枠に分離）
- 業界トレンド・他社事例まとめは**自社プロダクトの裏付けにならない**（補足解説枠か前提知識枠に分離）
- 隣接職種は隣接職種ルールに従って別行・別枠に配置（当該行に入れない）
