# カンファレンス企画フレームワーク — 方法論

> このファイルは詳細な背景リファレンス。日常の実行には `skills/` を使ってください
> （各フェーズに `skill.md` があります）。

## このフレームワークの位置付け

**技術／エンジニアリング系カンファレンス** を企画するためのフレームワーク。
最も難しい判断層 — **テーマ、ターゲット聴衆、登壇者リサーチ** に焦点を当てています。

物流（会場、ケータリング、スポンサー契約、登録）は対象外。

## 全体フロー

```
[Phase 1] 軸定義
   ↓
[Phase 2] Matrix A 作成（規模 × 目的）— claude.ai Deep Research
   ↓
[Phase 3] Matrix B 作成（職種 × 目的）— claude.ai + ハードゲート
   ↓
[Phase 4] コアテーマ抽出
   ├─ 4-a 機械的抽出（Active P-set 重なり）
   ├─ 4-b トピック起点ヒートマップ（各トピックを規模 × 職種で ◎○△×）
   └─ 4-c audience-shape クラスタリング
   ↓
[Phase 4.5] 内部検証ヒアリングループ（エンジニア 2〜4 名へ DM）
   ↓
[Phase 4.6] セッションテーマ構造化（大→中→小）— ADR-0005
   ↓
[Phase 5] 登壇者候補リサーチ（独自実装で拡張）
   ↓
[Phase 6] セッション設計／ヒアリング設計
```

各フェーズの時間目安：

- Phase 1: 30 分
- Phase 2: 半日（主に claude.ai に委譲）
- Phase 3: 3〜5 時間
- Phase 4: 2〜4 時間
- Phase 4.5: 2〜3 日（カレンダー時間；実作業時間 <2 時間）
- Phase 4.6: 2〜3 日
- Phase 5: 数日〜数週間
- Phase 6: 1〜2 日

## 目的軸（P1-P6）の定義

| ID | 目的 | 定義 | 典型例 |
|---|---|---|---|
| **P1** | 新規サービス立ち上げ | 0→1、MVP、PMF 探索 | スタートアップ全般 |
| **P2** | 既存サービスの拡張・スケール | 1→N、スケーラビリティ | 成長期 |
| **P3** | 複数サービスの統合・共通基盤化 | M&A、社内サービス統一、プラットフォーム化 | マルチプロダクト中〜大 |
| **P4** | 既存システムの刷新・モダナイゼーション | レガシー脱却、7R（AI に関係なく） | 非 IT 大手の基幹刷新 |
| **P5** | AI 機能の組み込み（Add-on） | 既存サービスに後付け、既存アーキは維持 | 「既存サービスに AI 検索追加」 |
| **P6** | AI ネイティブ化 | AI 中核に再構築 or 最初から AI 前提 | 「AI ファースト再考」 |

### P5 vs P6 の境界（最も混同しやすい）

| 観点 | P5 AI 組込 | P6 AI ネイティブ |
|---|---|---|
| 既存アーキへの扱い | 保持（API 追加・サイドカー的） | 再構築 or ゼロから |
| AI の位置づけ | 機能の 1 つ | システムの中核 |
| 失敗時の影響 | 既存機能は動く | システム全体が止まる |
| 設計時の思考 | "どう接続するか" | "AI を前提に全てを再考" |

## 活動レベル（◎○△×）

| 記号 | 意味 | 判定基準 |
|---|---|---|
| ◎ | 主戦場 | IR・中計で明示宣言、経営層が投資配分含めてコミット |
| ○ | サブ戦場 | 一部事業／部門で進行、または近年のプレスで主要発信あり |
| △ | 限定的 | 一部の試行・検討段階 |
| × | 該当少 | 事業特性上ほぼ発生しない |

## Matrix A の構築

実行スキルは `skills/phase-2-matrix-a/skill.md`。主要な方法論ポイント：

### ソース優先順位

1. IR・決算・中期経営計画（最も権威的）
2. 経営層インタビュー（PR、プレスリリース — 過去 12 ヶ月）
3. 採用ポジション・JD（現在の方向性シグナル）
4. 一般紙のテックニュース
5. 直近の経営層登壇・パネル

**技術ブログは補強証拠としてのみ**、Matrix A の主軸にしない。

### 時間軸

確立企業：2024-01 以降。
スタートアップ：2021 以降（カバレッジ限定的のため）。

### 出力フォーマット

標準 5 × 6 マトリクス。1 セルに：

- マーク（◎○△×）
- 一行根拠（ソースシグナルを指す 1 文）

参考 URL は規模セグメント単位で収集（セル単位ではなく、監査しやすさ重視）。

## Phase 4：コアテーマ抽出

### 4-a：機械的抽出

クロスカットトピック（複数の Matrix B セル — 異なる職種 × 目的 — に出現するもの）を探す。

### 4-b：トピック起点ヒートマップ

各トピックについて、規模 × 職種ごとに温度マーク（◎○△×）を割り当てる。
1 トピックにつき 1 ヒートマップ。トピックは通常 10〜16 のクロスカットテーマに落ち着く。

### 4-c：audience-shape クラスタリング

トピックを **聴衆形** で束ねる（同じホットゾーン）、トピック類似性ではなく。

典型クラスタ（あなたの領域ごとに生まれる）：

| クラスタ軸 | 結ぶ規模 | 例トピック |
|---|---|---|
| レガシー／規制 | 非 IT ↔ IT 大手 | 段階刷新、ベンダー責任、オンプレ回帰 |
| 共通基盤／統制 | IT 大手 ↔ メガベン | 共有 DB 脱却、自律 vs 統制、横断 ADR |
| 最初の専任化 | メガベン ↔ ベンチャー | 専任 SRE/Sec、SSO/OIDC 段階導入 |
| PMF 前・兼任 | ベンチャー ↔ スタートアップ | PMF 前最小構成、Make-Buy |
| AI 共通（横串） | 全規模 | AI 推論 SLO、AI Agent 新脅威 |

### クロスカットテーマ・ランキング

| 強さ | 条件 | 用途 |
|---|---|---|
| 最広射程 | 横 6 職種 × 縦 5 規模 | 基調軸 |
| 広射程 | 横 4〜5 職種 × 縦 4〜5 規模 | メイントラック |
| 限定射程・高熱量 | 横 2〜3 職種 × 縦 1〜2 規模 | サブトラック |

## Phase 4.5：内部検証

Matrix B は公開情報ベースの仮説なので、ターゲットに直接当てる前に内部実践者で
検証する。

### 実施手順

1. ターゲット聴衆の中から 2〜4 名のエンジニアを選定（職種多様性を担保）
2. DM 送信：目的軸定義 ＋ Matrix B 仮説抜粋 ＋ 「実感と合うか・ズレるか」質問
3. 回答の解釈：自己 P 帰属、実際の課題、確認・追加事項
4. パターンマッピング：構造が既知パターン（Data Mesh、Modular Monolith、Conway の法則 等）に
   一致するか観察。一致するなら第 2 ラウンド DM
5. Matrix B v2 に反映

### DM テンプレート（初回・追投げ）

`skills/phase-4-5-internal-validation/skill.md` 内のテンプレ参照。

## Phase 4.6：セッションテーマ構造化

実行スキルは `skills/phase-4-6-session-theme-structuring/skill.md`。
方法論ハイライト：

### サブステップ

1. 入力レビュー ＋ 焦点設定（キャッチコピー候補）
2. 大テーマの仮設定（3〜5）
3. **粒度の非対称性を許容**（戦略 ≈ 3、実装 ≈ 5-6、組織 ≈ 4）
4. 中テーマの軸選び（複数試行）
5. 小テーマは議論ポイント、ラベルではない
6. 批判的見直しループ ×2 最低
7. 抜け論点チェック（過去データ／隣接カンファ／Matrix B 集計）
8. 横軸 ＋ キャッチコピー最終化

### 中テーマの軸候補

| 軸 | 例 | 適する大テーマ |
|---|---|---|
| 時間 | 未来 / 現在 / 過去 + 未来 | 戦略系 |
| レイヤー | アプリ / データ / 運用 / インフラ / セキュリティ | 実装系 |
| 判断 | 広げる / つなぐ / 分ける | マルチプロダクト戦略 |
| 対立 | 自律 vs 統制 / 同期 vs 非同期 | 思想系 |
| フェーズ | 立ち上げ / 拡張 / 成熟 | 事業段階系 |
| 主体 | 人 / 決定論的 / AI | 責任分担系 |
| 階層 | マクロ / 中 / マイクロ / 個人 | 組織系 |

## Phase Z：再利用チェックリスト

新カンファレンスに本フレームを再利用する際の確認項目：

- [ ] 確認：新カンファレンスまたは大規模リブランド（年次更新ではない）
- [ ] 5 規模セグメントを領域に適応
- [ ] P1-P6（目的軸）を領域に適応
- [ ] 高優先 7 職種を領域に適応
- [ ] claude.ai プロンプトをカスタマイズ（Matrix A、Matrix B）
- [ ] ADR-0002/0003/0004/0005 の規律を継承
- [ ] 領域に応じて隣接職種ルールを調整
- [ ] 必要なら前提知識枠を活用
- [ ] 失敗パターン #1-20 を事前読み込み
- [ ] Phase 4.6（セッションテーマ構造化を一級ステップとして適用）
- [ ] 新しい失敗パターンが見つかれば文書化（Issue or PR）
